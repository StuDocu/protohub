Turn studying into a daily habit

It's the study intelligence — topics, gaps, the next review — that Course Companion creates and can surface anywhere; the course page is just its first home. Everything's ordered by time to exam with a daily practice loop on top that brings students back, scaling gracefully through the Data Confidence Levels.

Mission Anchor · Education AI  ·  Contributors: Edu Gregorio (PM, Data Confidence Levels + onboarding), Ömer Polat (design + FE), Akos Horvath & Ruby (eng), Tiago Avelar (data), Sri Majji (eng mgmt), Stephen Ballot (CPO)  ·  Infinite Practice: Ryan Papetti (canonical topics — RFC ↗) · Jo Huang (PRD ↗)

1 Problem

  • The Today page splits traffic across three competing sections — Recommended (44% of artifact clicks), Up next (32%), Start here (24%). No clear winner to keep, none to cut.
  • Start here is one-and-done (1.06 clicks/user) — orientation, not a reason to return.
  • We cut ~80% of document clicks by design, but never replaced the old "scan my docs" return habit with a new one.
  • We suspect lower retention vs. legacy but can't prove it: the test arm fires 0 Course View events (1.56M users all tag false), and control is the whole site, not matched courses.

Cost of not solving: the page converts in-session but never builds a habit → retention stays flat, and we can't even measure it.

2 The pitch

One surface, ordered by time to exam, with a daily review loop that builds a return habit.

The same experience works at every level of data we hold about a course — and we use Edu's L1→L4 Data Confidence Levels as the build phases, so scaling and shipping are the same plan.

3 Goals & Non-Goals

  • One surface, not three — collapse the competing sections into a single time-ordered spine.
  • A daily return habit — a finishable "review today" that gives a reason to come back tomorrow.
  • Scales to any course — works at every data-confidence level, no empty shells.
  • Trustworthy measurement — fix instrumentation so retention is provable.

Not in v1:

  • Full LMS / Canvas OAuth — Tier 3, later.
  • Cross-course aggregation — the "study OS" home is an L4 horizon, not v1.
  • Open-ended crowdsourced data — use validate-don't-contribute (confirm, don't author).
  • New artifact types — reuse mock / quiz / note; this is arrangement + a daily loop.

4 The concept — two clocks

Calendar axis

Temporal spine

What to study, and when. Days to exam.

Memory axis

Daily Practice Stack

What to review today. Decay & gaps.

They share the same data and never compete on screen: the spine plans the term, the daily stack runs the habit, and the spine's dates feed the stack's priority. Topics (radar, readiness, gaps) ride inside the daily card and as quiet tags on spine rows.

💡

Key insight: the daily engine has a data source no pipeline can shortcut — the student's own practice history. It only sharpens through use, so it bootstraps itself. L1's whole job is to make the first session frictionless.

5 Data Confidence Levels draft — for alignment w/ Edu

How much do we trust what we know about a course — and how honestly do we signal it?

The surface is always populated. What changes per level is how confident the recommendations are and how openly we flag uncertainty — never an empty page, never a guess shown as fact. Two data tracks move independently:

Shared · course-level

Dates & topics

Estimated, then validated. Confirmable by anyone — no account needed.

Personal · per-student

Practice history & mastery

Only the student generates it. Needs an account — powers L3+.

L1Estimated
What we holdNothing course-specific.
How we signal itBest-guess, clearly marked est. Popular-material warm-ups, estimated/locked timeline. Useful with zero data and zero setup. Works logged-out.
L2Enriched — estimated + community-validated
What we holdAgents extract likely dates + topics (estimated).
How we signal itShown as est until validated; then the tag drops. Still works logged-out.
👥

Crowdsourcing dates (L2) — agents seed the guess, students validate it ("validate-don't-contribute", opportunities #19).
• ≥2 active-in-course students confirm → community-confirmed, drop the est tag.
• Anyone can flag / correct; a correction with a specific date weighs more than a passive "looks right" tap.
• ≥5 active students converging on the same alternative date → switch.
• A syllabus / LMS-backed date (#19 pipeline, legal per Sam) outranks votes — instant confirm, no crowd needed.
• Confirmations are term-scoped and expire; multiple legit dates (sections) allowed, not forced into one.
• Only students active in that course vote, one each.

L3Confirmed + personal
What we holdConfirmed date (source or community) and the student's own practice history / mastery.
How we signal itReal readiness %, topic radar, gap detection, spaced "3 to review today." Mounts on the personal / exam-readiness data the readiness team is building.
RequiresAn account — personal data has to persist.
L4Fully personalised — this course
What we holdConfirmed dates + rich personal history + the student's own uploaded materials.
How we signal itZero-friction, fully tailored within the course; precise "come back tomorrow."
🔐

Progressive auth — don't wall the front door. Logged-out students get L1–L2 (all shared data). The sign-in prompt fires after investment — once they've answered the warm-up quiz or finished a daily card — because that's when there's personal history worth saving and a real reason to sign in. The auth wall is the L2→L3 boundary, triggered by engagement, not a gate before content.

🧭

Resolving "what is L4?" The Linear router currently reads L4 as cross-course + peers; Edu's definition reads it as this course, fully known. Proposed resolution: the L1–L4 ladder measures one course's confidence. Cross-course interleaving + peer signals are a separate axis — the "study-OS" horizon (already P2, "design for, don't build"), not L4. Needs Edu + Ruby sign-off, since the router gates L4 = cross-course today.

AB Two layouts, one daily card

The daily card is fixed. What comes after it is an A/B we switch by context.

Default

A · Materials feed

Daily card → our artifacts & materials directly, reason-tagged (the “hero + feed” layout). Familiar, conversion-safe, works even when topics are thin. Best at L1–L2.

Topic-forward

B · Topic spine

Daily card → the time-ordered spine, every row topic-tagged. Leans into Infinite Practice. Best when topics + mastery are rich (L3–L4).

Default to Materials feed, graduate to Topic spine as topic confidence climbs — or A/B them head-to-head on D7 return. Same daily card in both, so the test isolates the “after” arrangement without touching the habit engine.

7 Requirements

P0 · must ship
  • Spine renders buckets from a single bucket(artifact, dates) function.
  • Daily card with three states: front → question loop → result.
  • Confidence is an input — lock (L1), estimated (L2), confirmed (L3) from the same render path.
  • Reason / bucket param on every artifact click (don't lose attribution when sections merge).
🚧

Blocking pre-req: the test arm must fire a shared return event with testSegmentNext=true. Nothing below is measurable until this lands.

P1 · fast follow
  • FSRS spaced scheduling; gap detection; topic radar + ReadinessBar.
  • "Add / Confirm date" → writes to the confidence-scored field schema.
P2 · design for, don't build
  • Cross-course aggregation; LMS OAuth; peer-shared content surfacing.

10 Risks & mitigations

  • Empty radar reads as "broken" (L2) → consider a one-liner instead of an empty chart; usability-test it.
  • Wrong estimate erodes trust → always show est, confirm in one tap, never present a guess as fact.
  • Crowdsource locks a wrong date → a passive "confirm" is low-confidence; weight specific-date corrections higher, let a syllabus/LMS source override votes, scope confirmations to the term.
  • Too many buckets re-fragment → cap visible buckets at 3 + "Later".
  • Post-midterm dead zone → fill the lull with spaced review, or retention dips right when we'd want re-engagement.
  • Cognitive-load creep → one primary action per screen; sparse chips; topics emerge with confidence, not all at once.
