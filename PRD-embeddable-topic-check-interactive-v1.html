Drop a check where students already study

One embeddable URL — a course/topic-scoped practice check — that a student pastes into their own Notion page or a shared Google Doc, where the next classmate can't help but try it. It's a detachable piece of Anchor: Infinite Practice supplies the questions, the Course Companion is where the CTA lands, and the paste itself is the sign-up event. No SEO, no ads-to-survive, no peer-social system to maintain.

Mission Anchor · Education AI  ·  Proposed by Ömer Polat  ·  Connects to Infinite Practice (canonical topics + question widgets) and the Course Companion  ·  Event: The Hackquisition (24–26 Jun build · seed to 3 Jul)

1 Problem

  • Studocu's acquisition is overwhelmingly search-dependent, and the channel is eroding: Google's Sept-2025 num=100 change already scrambled impressions, and AI-mode is rising. The Hackquisition rule bans an SEO-reliant loop outright.
  • Even the app channel is rented: ~54% of installs come from app-store search, ~50% of installs are branded organic — i.e. people who already knew us. We own almost no net-new discovery surface.
  • Students live in tools we have zero presence inside: interview synthesis repeatedly shows them storing and accessing study materials in Notion, Google Drive, and the LMS; one recurring participant uses Notion AI specifically because it already holds her notes.
  • The fast, viral fix — student-to-student social systems — is a path we've been burned on before and explicitly don't want to rebuild.

Cost of not solving: we keep renting Google (and the app stores) for users while the rent goes up — with no owned channel of our own as search fragments into AI.

2 The pitch

One embeddable check, dropped into the note app a student already uses, that the next classmate can't help but try — and a try ends in a sign-up.

It rides the student's existing note-sharing instead of building a new social network: a student adds it to their own study page because it's a useful self-test, and every co-editor or viewer of that page meets a live, gradeable question with a "make your own → sign up" exit. The questions come from Infinite Practice's canonical topics; multiple-choice is graded in the browser (no auth to try); the CTA deep-links into the live Course Companion.

3 Goals & Non-Goals

  • An owned, non-SEO sign-up loop — discovery that doesn't route through Google or paid.
  • Demoable Fri 26 — a clean stage moment: paste a link, it renders live, a viewer signs up.
  • Durable to Jul 3 and beyond — keeps pulling sign-ups async; real payoff is back-to-school.
  • Reuse, don't rebuild — stand on Infinite Practice's topics + question widgets and the existing Course Companion.

Not in v1:

  • Any two-sided / peer-social mechanic (share-to-unlock, invite-to-see). The spread rides a student's own placement, full stop.
  • Live on-the-fly generation inside the embed — use a pre-generated / cached bank; generation latency kills the "instant try."
  • Google Workspace Marketplace add-on / Notion gallery listing — real distribution, but OAuth + review runs weeks; can't prove sign-ups in the window.
  • New question types — reuse the existing widget model (MCQ / fill-in-the-blank first).

💡

Key insight: the artifact is useful to its owner first (a real self-test on their topic), so they place it for their own reasons. 7 Requirements

P0 · must ship
  • An embeddable page that renders a question-widget JSON payload and is iframe-able (permissive frame-ancestors so Notion can embed it).
  • Client-side grading for MCQ / fill-in-the-blank — a guest can answer with no auth, no round-trip.
  • A CTA → live Course Companion deep-link carrying an attribution param (so a sign-up is traceable to the embed + topic).
  • OG / oEmbed metadata so a pasted link unfurls richly (Docs chip, chat preview, Notion bookmark).
🚧

Blocking pre-req for scale (not for the mock demo): a guest-readable, cached, course/topic-scoped question source. Today questions are generated on the fly and scoped per project + per user — an anonymous viewer has none. Phase 0/1 sidestep this with a static bank; Phase 2 needs the Infinite Practice team to expose a cached bank.

P1 · fast follow
  • Pre-generation pipeline: run question-gen offline for the seeded topics → static bank (piggybacks Infinite Practice's bulk-extraction-to-S3).
  • Per-topic OG image generator (the chat/Docs preview).
  • Bot / rate-limit handling on the public embed endpoint (we've fought this on course pages before).
P2 · design for, don't build
  • Guest-readable cached bank (the durable dependency above) — design the contract now, build when IP's ready.
  • Workspace Marketplace add-on / Notion gallery listing — real channel, too slow for the window.
  • Open / short-answer grading via the feedback agent — MCQ-only keeps the guest path zero-backend.

10 Risks & mitigations

  • Peer-sharing creep (the thing we got burned on) → keep it strictly one-sided: a student places it for their own use; no invite/unlock mechanic. The loop is "your own note page," not a network.
  • Post-finals timing kills the Jul-3 number → seed summer / resit / quarter-system / southern-hemisphere courses; frame Jul-3 as directional, real payoff is the Aug back-to-school window (what the €5k prize is for).
  • No guest question bank exists → Phase 0/1 use a pre-generated static bank; only Phase 2 needs the cached source from IP.
  • Topics interim + syllabus-only → small Anchor-live ∩ topics intersection today → seed exactly that intersection; don't over-promise breadth.
  • Notion/CSP blocks the iframe → test the Notion embed on day one; ship oEmbed + a hosted fallback page.
  • Generated-question quality / cringe → human spot-check the seeded bank before it goes into real student pages.
  • Building embedding twice → check whether to ride Exam Ready's Mini App primitive before building a bespoke host.

11 Open questions & dependencies

  • [Product] Mock demo or real APIs for Fri 26? The §6 decision. (decide first — everything else forks here)
  • [Eng/Data] Can IP expose a guest-readable, cached, topic-scoped question bank? Today it's on-the-fly + per-project/per-user. (blocking scale — Infinite Practice)
  • [Data] Which Anchor-live courses already have canonical topics generated? Defines the seed list. (blocking seed — Anchor)
  • [Eng] Can the check reuse the Mini App sandboxed-iframe + shareable-URL primitive instead of a bespoke embed host? (Infinite Practice / Exam Ready)
  • [Eng] Embed host: iframe frame-ancestors policy + oEmbed endpoint + bot handling — who owns it? (Anchor eng)
  • [Data] Topic model: course-level canonical vs project-level (an open data-model question) — does the embed's reliance on a stable course-topic ID hold? (Infinite Practice)
  • [Design] The CTA wording + the post-try sign-in moment — what's the smallest ask that still converts a guest?

Dependency chain: 1) decide mock-vs-real (§6) → 2) seed list = Anchor-live ∩ topics (Anchor) → 3) pre-gen static bank for those topics (Infinite Practice) → 4) embed host + oEmbed (Anchor eng) → 5) attribution event Signup From Embed (Anchor data) → 6) durable cached bank (Infinite Practice) unblocks scale.
